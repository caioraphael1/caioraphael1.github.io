About

  • LDtk .

  • LDtk .

  • Open-source.

  • Made in Haxe.

  • "From the director of DeadCells".

  • Formerly 'LEd'.

Impressions

  • Initially the app is intimidating with the amount of information, giving the impression that it is quite bloated. However, if you ignore the examples and create a blank project, the app becomes much simpler and more pleasant.

  • There is no customization of shortcuts or interface customization.

  • The button to create an entity, resize an entity and select an entity is the same (MB1) and cannot be changed.

    • This is annoying, many times I create an entity when I only wanted to move the existing one.

  • The exported data is considerably more complex than Tiled's data.

    • This is not necessarily a problem, since it is still a json file.

  • [2025/03/06]

    • The issue tracker  has several bug reports, but it is quite abandoned.

    • There are basically no PRs.

    • Both Issues and PRs receive no response or comment.

    • The last update was 1 year and 2 months ago.

    • The last commit was 4 months ago.

Compile

Setup

  • haxe setup.hxml

    • Install required Haxe libs, listed in the file setup.hxml .

  • cd app  -> npm i

    • Install Electron locally and other dependencies through NPM.

Compile

  • haxe main.debug.hxml

  • haxe renderer.debug.hxml

  • cd app  -> npm run start

  • The app/package.json  contains a list of other commands to use with npm run _command_

Change entries

  • In LDtk :

    • Add the entry in the class by changing the entries in:

      • new

      • toJson

      • fromJson

    • Visually:

      • related .html  file.

        • Makes things appear in the interface, but nothing is saved when making any change.

      • ui/modal/panel/_related_script_/updateForm()

        • Used to save the changes made in the interface. Without this, opening and closing a tab in the app loses the information.

        • References:

          • zindex .

          • ysort .

          • colWidth .

          • colMode .

  • In ldtk-haxe-api (Schema) :

    • About :

      • It is necessary to change the ldtk-haxe-api , because it is there that the json schemas are obtained.

      • The classes ldtk.Json , etc, come from that lib.

      • It is not necessary to change docs/JSON_SCHEMA.json  or docs/MINIMAL_JSON_SCHEMA.json .

      • It is not necessary to change the version of ldtk-haxe-api  before running the haxelib git ldtk-haxe-api ... etc  command.

    • Change the files of the ldtk-haxe-api .

      • I believe the only important file is src/json.hx

    • COMMIT the modifications to the remote.

      • You need to do this every time you change the ldtk-haxe-api .

    • Run LDtk - Fetch Schema .

      • Runs this in the background: haxelib git ldtk-haxe-api https://github.com/caioraphael1/ldtk-haxe-api.git caio

      • You need to do this every time you change the ldtk-haxe-api .

  • Finally :

    • Run LDtk - Compile & Run .

I figured it out: I had to fork https://github.com/deepnight/ldtk , make my changes in the src/ldtk/Json.hx  file, push to remote, and run inside the ldtk root folder haxelib git ldtk-haxe-api https://github.com/caioraphael1/ldtk-haxe-api.git caio , which refers to my github fork and branch caio .

Dream Tool

  • .

Current needs

  • Polygonal collision.

    • For any tile or entity that I want a custom collision.

    • Problems :

      • Tiles with polygonal collisions are not possible, and I do not want to implement them in Haxe.

    • Ideas :

      1. Do not define collisions in the tiles or tileset, but rather in a separate IntGrid.

        • Ends up generating tons of extra collisions, because it usually takes a "tile" from the IntGrid with reduced size.

          • This can be optimized later in Odin, but it's meh.

      2. Do not define collisions in the tiles or tileset, but rather in instantiated Entities.

        • Can technically be more optimized than the IntGrid, in terms of generated collision regions, but it's not worth it because the entity editor is so bad.

        • The entity editor is terrible.

          • Cannot delete or move things in bulk.

          • Placing and removing entities is painful and often fails.

      3. ~Use Enums to mark tiles that should have their ENTIRE area as a square collision, while ignoring the idea of having different and individual collisions.

        • It's obvious this is a limiting design.

      4. ~Tiles marked with an enum are translated to have a type of collision in-game.

        • This requires multiple options for the enum.

          • "Top fence", etc, etc, etc.

        • It's super annoying and limiting that every time I want something new I have to define an enum, etc, etc.

        • It's confusing. It's ugly. It's not very customizable.

      5. ~Use tiles' "extra metadata" and write some custom entry that would represent what I want.

        • Although it could be more specific with metadata, there is no good visual representation of what the metadata is within the tile, and I am not sure I got the syntax right for how I should write it.

        • Parsing this data is also annoying, because there is no coherence and I always have to read all tiles considering there might be some unmarshal.

          • Actually the data is stored in an array of dictionaries, so that spares the unmarshal, not making much difference.

          • Still, there is no certainty if the syntax was correct in LDtk.

      6. ~I don't store any metadata in LDtk and just translate everything in a single pass in Odin, by associating the "tile number+image name" with which characteristics that tile should have.

        • It's very strange to have no context of WHICH texture and WHICH idx should have X characteristics.

  • Pillars with ysort.

    • For anything I want to pass behind.

    • Problems :

      • Considering there are no YSort options anywhere, and there are no options for individual tile Origins, YSort must be done via some workaround.

      • The fact that tile_size and grid_size must be square also hinders, by forcing sprites to be adapted for pillars, etc.

    • Ideas :

      • YSort and Pivot.

        1. ~Use the feature I implemented where each Tileset has a universal PivotPoint, so that all tiles have their pivot placed in the same spot.

          • Use the Tileset Pivot as the texture origin point.

            • This would be done exactly like this in the future, but with customizations so that each tile would have its own "origin" and also entries for "ysort offset"; in other words, in the future this would be similar but with MUCH more customization.

        2. ~Use tiles' custom metadata to indicate what the tile's "ysort origin" is.

          • Creates annoying parsing problems, etc.

      • Tile_size and grid_size:

        1. ~Ignore the tile_size and grid_size problem by adapting all sprites to be square.

          • It's very annoying and uncomfortable having to do this.

        2. Implement the feature in LDtk.

  • Positioning of point entities.

    • For spawning characters, creatures, items, etc.

    • Problems :

      • There is no clear definition of how entities should be used, so this intersects with Images and generates confusion.

    • Ideas :

      1. Make any entity with size 1, or with a larger size and ignore the region. Either works.

  • Background images.

    • For moments when I want to draw the entire scene without tiles and do collisions afterwards.

    • Problems :

      • There is no definition for this. It's all done via a workaround with Entities and Visual Editor as image. It's terrible.

    • Ideas :

      1. Provide official support for Images in Entities.

        • Without using a tileset.

        • Clear separation between: Editor Visual and Ingame Visual.

      2. Continue using the current workaround, where the Editor Visual's selected tileset is used at its maximum size in Odin.

        • This is very bad, incoherent, crap, confusing, and awful.

Current problems

  • Square tile_size.

    • Interferes with pillars, etc, many things.

  • Tile_pivot is not per tile.

    • Causes issues when there are different things in the tileset with different pivots and ysorts.

  • Tiles do not have metadata.

    • Hinders defining collisions, occluders, etc.

  • No official support for Images.

    • Must use Tilesets.

    • Must use Entities, with no separation between editor visuals and ingame visuals.

Things I will never implement

  • Animations.

  • Patterns.

  • Polygons and other geometries.

  • Metadata with geometries (for Entities, Images and individual Tiles).

  • Pivot for individual Tiles.

Tileset

  • Def :

    • Source-regions in a texture, placed in a grid; where each region contains metadata.

  • TileSet tab of Godot's TilemapLayer, but changing the design presentation.

  • Things Godot did wrong:

    • The textures tab is stupid. It could be a list of textures loaded in the tool.

    • ID doesn't make sense.

    • Atlas_Coord and Atlas_Size are very confusing and do not allow 2 tiles to draw things that intersect.

      • In practice, Godot allows a tile to not be truly a "tile" by extending outside the texture division region. It's a confusing region system.

        • Honestly, I don't know how to make the system less confusing.

        • Intrinsically it is confusing because it betrays the cropping done from the texture using the 'grid_size'.

        • I thought of making the Tileset have a list of "regions" defined, where each region can be a classic tile, or a large Godot "tile" that corresponds to a custom source region.

      • Also, I don't think those custom regions should have the Terrain option.

      • Either way, the system is quite interesting and I used it often to help with tiles of different sizes being created in the same TileSet.

    • The Setup tab should contain the Terrain information.

    • The Tile->Terrain tab is stupid.

      • The Setup tab should contain the definitions of all Terrains.

      • Besides, editing the terrain bit of each tile sounds odd. I don't think each tile should have the Terrain bit data; it sounds weird and should never be used externally, it's nonsense.

    • Tile->Z_Index.

      • This can normally be controlled by the order of Layers.

  • Optional and/or unnecessary Godot things that should be done only on demand:

    • Tile->PhysicsLayer (collision polygon).

    • Tile->OcclusionLayers (occlusion polygon).

    • Tile->Probability.

    • Tile->Modulate.

    • Tile->Material.

    • Tile->PhysicsLayer->LinearVelocity.

      • Better placed in a "physics properties" metadata entry.

    • Tile->PhysicsLayer->AngularVelocity.

      • Better placed in a "physics properties" metadata entry.

    • CustomDataLayers + CustomData.

      • Totally meh. It's better to actually create the desired entry than to create a custom data.

  • Things LDtk did wrong:

    • LDtk makes the Pivot Point belong to the entire TileSet, instead of being per tile (Godot: Rendering->TextureOrigin).

    • AutoTiling should be associated with the TileSet, not the Layer.

    • Tile dimensions, spacing and padding are square, when there should be the option to define X and Y separately.

    • No control of YSort origin.

    • The pivot point should be pixel-based, not %.

  • Things LDtk did ok:

    • Enums are a way to "paint a tile with metadata", so that's fine.

    • Tags for Tilesets are nice in theory for helping with folder separation (only), but the way it was done is very confusing. It should simply have an option to create folders and drag the TileSet into that folder.

      • Either way, this option is very silly and only organizational.

Entities

  • Def :

    • Dynamic regions in the world containing metadata.

  • LDtk Def :

    • Dynamic regions with visuals ONLY for the Editor.

      • * The Problem :

        • Practical :

          • There is no clear definition of how to spawn the INGAME visuals of that region. Is the region just an Area? A Trigger? Or should it represent a real, tangible ingame visual object? If Entities are used only as invisible regions, then what is used to spawn the real visuals? That doesn't exist.

            • Ex : Consider a door. I want the door's visuals and an area for proximity to auto-open the door.

            • The door needs ingame visuals. The area does not need visuals. The area is larger than the door, obviously, as a proximity trigger.

            • I can create an "area trigger" Entity that is invisible for the door.

            • But what is the door? I have no idea.

              • If it's a tileset, the "area trigger" Entity CANNOT link to tiles spawned in the world; an Entity can only reference Entities or Tiles WITHIN a tileset (that is, only a source region in a texture).

              • What other options exist? None. The door HAS to be an Entity, when clearly Entities were not made for that.

                • Remember that the door asset therefore needs to be ANOTHER entity, besides the Area Trigger entity, because they have different sizes.

                • However, to be clear, it's terrible to use an Entity to spawn a texture in practice.

        • Using Entities for Textures :

          • The Entity's size does not adjust to the texture size. You must use several workarounds to make it happen.

          • Editor Visual is the interface used to select the texture, but in the Entity DEFINITION, this is only a visual preview, not what is actually used ingame; at least it wasn't designed for that; it's counterintuitive and weird.

  • Inspired by Trenchbroom.

  • Purely visual LDtk niceties:

    • Entity Opacity (different from the layer's overall opacity).

    • Entity Line.

      • By default the line should be 100%.

    • Editor Visual (texture and region color).

    • Alt UI Icon.

    • Entity Fill.

  • Things LDtk did wrong:

    • Selecting an entity turns it yellow and you can't see anything.

    • Hiding the ORIGIN POINT of an entity is wrong. It's very confusing to deal with point entities and try to figure out where things will actually happen.

      • Correction: the origin point is displayed, but by default the Editor Visual and Fill drawing are in front, covering everything.

    • Pivot Point in percentage is odd when it comes to something that can be resized.

      • It's good to have a Pivot, seems okay, maybe.

      • However, in this case I really think it should be a pixel-based pivot point, not %.

      • Maybe the texture should be a region completely detached from the entity's region.

        • I believe the texture should inherit the position of the entity region. I don't think it's necessary for them to be "siblings" in the hierarchy.

Dissatisfactions

Editor

  • Does not support placing images in the world well.

    • It is always necessary to use a TileSet.

      • Use a tile that composes the entire image, or multiple smaller tiles that together compose the entire image.

  • LDtk's YSort is not done across different layers.

    • Severity :

      • It's just a visual inconvenience in LDtk, it doesn't break the data.

    • "A layer with YSort enabled should interact with the tiles contained in itself and with other layers that also have YSort enabled".

      • In other words, it's like adding itself to a pool to perform ordering.

  • LDtk changes casing and removes characters, even in the most "liberal" case.

  • There is no grouping of Layers.

    • This would be useful to organize the Floors of my game.

    • Not gamebreaking or anything. I use IntGrid to make the separation.

  • Does not allow Isometric or Hexagonal Grids.

Entities

  • Entity placement tool :

    • EntityLayer entities cannot be flipped.

      • Tiles can be flipped with the X or Y button, but entities cannot.

    • Does not allow deleting entities in bulk with Shift MB2.

    • Does not allow selecting entities in bulk with Shift MB1.

  • Entities are not drawn with YSort.

    • I felt this was a problem when drawing trees as Entities.

Tileset

  • tile_size is always square; the TileSet does not allow editing tile width and height.

    • I felt this was a problem when drawing the cliff_32x16x16.

      • This made me have to edit the tile to be 48x48 instead of 16x48.

  • Padding and spacing of tiles are also square, not per axis.

  • Collision definition :

    1. ~ Each tile has a metadata field that accepts a polygon to represent collision.

    2. The TileSet has an Enum with 3 fields; each field shows the following options below when selected:

      • Impressions :

        • [2025/06/11] Does not solve basically any problem. Works for some extremely specific cases, but it's not a good solution.

      • "I place the pivot at the base of the tree trunk and associate a capsule-shaped collision enum".

      • Box.

        • width.

        • height.

      • Capsule (box with trimmed edges).

        • width.

        • height.

      • Circle.

        • radius.

    3. The TileSet has a single collision editor, so you can draw 1 polygon that represents the collision for ALL tiles.

  • Pivot Point :

    1. ~ Each tile has its own custom Origin (former pivot-point), measured in px, not %.

    2. It is necessarily a TileSet characteristic.

      • Impressions :

        • [2025/06/11] Although useful, it's much better to have a pivot associated with each tile, instead of the tileset, because it's annoying to deal with varied tilesets.

      • This makes it consistent with Entities.

      • This causes only a single Pivot editor to be implemented, valid for:

        • TileLayer.

          • Makes total sense, since TileLayer can only have 1 TileSet anyway.

        • InGrid / Auto-Layer.

          • Doesn't even have a PivotPoint, it makes sense to have it on the TileSet since IntGrid also can only have 1 TileSet.

    • It is not allowed to specify the tile's origin point.

      • This can be solved in IntGrid or in TileSet.

        • I must evaluate which of the two options makes more sense, considering TileLayer, IntGrid AND Auto-Layer problems.

      • This ends up being a bigger problem for Tibia-style cases.

      • It's also annoying when you want to place a simple decoration element, but it occupies a non-conventional space in the tile.

        • It's just an inconvenience, but not a big deal.

      • It is not possible to specify the YSort origin point of each tile.

        • This is solved with the ability to change the tile's origin while using that origin as the YSort Origin.

      • Useful :

        • The EntityLayer allows a nice customization of the PivotPoint; I should evaluate if that is valid for other cases.

        • The TileLayer allows changing the origin.

        • IntGridLayer and Auto-Layer do not allow changing the origin

          • This hinders drawing autotiles.

          • I felt this was a problem when drawing the cliff_32x16x16.

IntGrid

  • IntGridLayer tiles cannot be flipped.

  • IntGridLayer's YSort is buggy.

    • This is the YSort of a single layer with itself.

    • It's clearly buggy, draw the image below and play with it to understand.

  • RulesWizard flips tiles automatically, greatly disrupting setting different tiles.

TileLayer

  • Tile preview for TileLayer is quite confusing, resulting in a strange offset.

  • When flipping a Tile on any axis, it's confusing where the tile will be placed.

    • If the Tile had a preview like the EntityLayer this would be much better.
