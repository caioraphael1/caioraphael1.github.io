RayLib

  • RayLib .

Features

  • Designed to be easy to learn and use. Has a friendlier and more direct API, especially for beginners.

  • Includes 2D/3D rendering, audio management, user input and basic file handling. Focused on simplicity and 2D games.

  • Optimized to be lightweight and efficient, but may lack advanced optimizations compared to SDL.

  • Windows, Linux, macOS, Android, Web (via Emscripten), and some embedded platforms.

  • More popular among indie developers and beginners.

  • Apparently uses integrated graphics by default .

  • Drawbacks and Limitations :

    • Graphics:

      • Uses OpenGL 1.1 (or 3.3+) under the hood; lacks modern rendering features like compute shaders, PBR, or GPU-driven pipelines.

    • Assets:

      • You must handle asset loading and hot-reloading manually.

    • 3D:

      • Very basic 3D support (e.g., limited lighting, no skeletal animation); not suitable for high-end 3D games.

    • Audio:

      • Based on miniaudio; limited advanced features (e.g., no built-in reverb, spatialization is basic).

    • Input:

      • Simplified input API; lacks fine-grained control for multi-touch, IME, or raw input.

Graphics

  • rlgl .

    • "An abstraction layer for multiple OpenGL versions (1.1, 2.1, 3.3 Core, 4.3 Core, ES 2.0, ES 3.0) that provides a pseudo-OpenGL 1.1 immediate-mode style API (rlVertex, rlTranslate, rlRotate...)"

    • It is an ugly OpenGL abstraction by the RayLib author, contained inside RayLib.

  • Simplified high-level OpenGL wrapper

  • Mainly OpenGL (2.1+ / ES2), via its own wrapper

  • Will RayLib support Vulkan? Probably not .

  • Has built-in support for 3D graphics using OpenGL.

  • No explicit multi-backend support; uses fixed OpenGL by default.

  • High-level 2D/3D API, simplified

  • Shader customization, but depends on OpenGL shaders.

  • Render pass control is not directly exposed; framebuffer is abstracted.

Audio

  • Full support for sound and music with an abstraction

  • Backend

    • Uses miniaudio

  • Abstraction

    • High: supports music, effects, volume and pitch control

  • Complete audio system

    • Yes: loading, playback, stop/pause, effects

  • Dynamic allocation

    • Yes.

  • Supported formats

    • WAV, MP3, OGG, FLAC (via stb_vorbis/miniaudio)

Input

  • Polling based on GLFW

  • Backend

    • GLFW (Windows/macOS/Linux), WinAPI in manual builds

  • Keyboard and mouse

    • Yes (via polling IsKeyDown() , etc.)

  • Gamepad

    • Yes (with support for multiple gamepads via GLFW)

  • Touch

    • Yes (basic multitouch input)

  • Event system

    • No (polling via functions)

  • Cursor and window

    • Full control via RayLib API

Sokol

About

  • Sokol .

  • sokol-odin .

  • Sokol tutorials for Odin .

    • Code produced at the end of the 4 episodes .

  • Written in C, not C++.

  • The core headers are standalone and can be used independently from each other.

  • Dependencies :

    • None.

  • Drawbacks and Limitations :

    • "Very low-level, more like SDL or DirectX than RayLib".

    • Being header-only and minimal, customization requires writing additional layers; not ideal for rapid prototyping.

    • Audio:

      • SokolAudio is extremely minimal; only supports basic buffer streaming (no mixer, 3D sound, or effects).

      • Basic audio playback (no recording).

Graphics

  • Support:

    • Direct abstraction over: OpenGL 2/3, D3D11, Metal, WebGPU, no Vulkan .

      • No Vulkan support .

      • Sokol doesn‚Äôt support Vulkan directly, but you can use it for window/input and pair it with Vulkan.

      • No Vulkan helpers‚Äîyou‚Äôre on your own.

    • Backend specified at compile time.

  • Pipeline abstraction, explicit commands (buffers, passes, shaders).

  • Approaches modern graphics APIs (WebGPU-style).

  • On the graphics side, it‚Äôs an abstraction close to using OpenGL or WebGPU; you still need to write shaders and set up GPU buffers, but you gain control over how things work internally.

  • Shader customization GLSL/HLSL/MSL according to backend.

Audio

  • "On the audio side, its audio interface only handles taking a stream of processed audio data and sending it to the appropriate audio interface. That means it requires you to load your own audio files, and process/mix the audio yourself."

  • PCM streaming only

  • Backend:

    • WASAPI (Windows), CoreAudio (macOS), ALSA (Linux), WebAudio

Input

  • Event-driven, directly from the OS

  • Backend

    • Native (Win32, Cocoa, X11, etc.)

  • Keyboard and mouse

    • Yes (via callbacks)

  • Gamepad

    • Partial (some backends)

  • Touch

    • Yes (with manual support on mobile platforms)

  • Event system

    • Yes (event-driven)

  • Cursor and window

    • Basic control (size, visibility, etc.)

Platforms

  • Adaptation :

    • sokolapp.h  provides complete abstraction of window, input and main loop. You don't need to implement platform-specific code.

    • But different from SDL or RayLib, Sokol requires you to provide callbacks for events, in a more explicit style.

  • Support :

    • Windows

      • Win32, D3D11, GL

    • Linux

      • X11, GL

    • macOS / iOS

      • Metal, Cocoa

    • Android

      • NDK

    • Web

      • Emscripten, WebGL or WebGPU

    • Raspberry Pi / Embedded

      • Partial (via GL or customizations)

  • Use of SDL3 in Godot for the input system .

    • Initial proposal .

About

  • SDL .

  • SDL .

  • SDL in Odin .

    • Code at the end of 14.2 episodes .

  • It is a multimedia library that handles aspects like graphics, audio, user input and window management.

  • It does not handle more complex aspects like physics, AI, or entity management.

  • SDL 2.0 and SDL 3.0 are written in C.

    • It is 100% procedural C, no classes, inheritance, namespaces, etc.

    • The library can be used from C++.

  • Drawbacks and Limitations :

    • 3D:

      • SDL3 doesn't include a 3D renderer like the 2D SDL_Renderer. For 3D you must learn either SDL_gpu, which is not as simple as SDL_Renderer, or use a proper graphics API.

      • CategoryGPU .

    • Audio:

      • Provides a low-level, push-based audio system. For mixing, spatialization, or effects, external libraries (e.g., SDL_mixer or miniaudio) are needed.

Graphics

  • SDL3 provides window creation ( SDLVulkanCreateSurface ) and input handling, but no Vulkan rendering helpers.

    • You write the Vulkan code yourself (swapchain, pipelines, etc.).

Characteristics

  • More flexible and powerful, but its API can be more complex, requiring more code for basic tasks.

  • Alone, it does not provide native 3D support.

    • Must be combined with OpenGL, Vulkan or another graphics API.

  • Commonly used in professional engines, with more optimizations for performance and platform compatibility.

  • Used in commercial games and engines like Unity and Unreal for input and audio support.

  • Broader platform support, including consoles like Switch, PlayStation and Xbox, and mobile platforms.

  • Focused on window management, input, audio and graphics, but typically requires OpenGL, Vulkan or other libraries for 3D.

Platforms

  • Adaptation :

    • One main goal is to abstract the platform layer , avoiding platform-specific code by the developer.

    • RayLib and SDL already handle the platform layer; you don't need to write one.

    • No per-platform adaptation needed except for advanced cases (e.g., native Vulkan or Metal code).

  • Support :

    • Windows (x86, x64, ARM)

    • Linux (X11, Wayland)

    • macOS / iOS (via Metal or OpenGL)

    • Android

    • FreeBSD

    • Web (via Emscripten)

Changes in SDL 3.0

  • Removal of redundant SDL_  prefixes.

    • Ex: SDL_RendererInfo  ‚Üí RendererInfo

  • More logical grouping of functions and types.

    • Interfaces are more consistent and predictable.

  • Modularization

    • SDL was split into separate modules.

    • Allows loading only what's needed, reducing size and build time.

  • Better support and integration with:

    • Vulkan, Metal, Direct3D 12

    • Wayland

    • UWP and modern consoles

  • CMake is now the main build system.

  • Types were unified and organized by namespaces.

    • Ex: SDL::Window , SDL::Renderer

  • Not compatible with SDL 2.x.

bgfx

Impressions

  • I would not choose bgfx because it's in C++ and I saw a video about the framework being very confusing.

About

  • bgfx .

  • bgfx - Documentation .

  • bgfx is a rendering abstraction library , not a windowing/input library like GLFW, nor a full game framework like RayLib.

  • It provides a uniform rendering API across multiple graphics backends (e.g., Vulkan, Direct3D, OpenGL, Metal).

  • You write rendering code using bgfx's API, and it internally maps that to the appropriate backend.

  • It supports runtime backend selection and automatic fallback if a backend (like Vulkan) is not supported on the current platform.

  • Cross-platform.

  • Can be used with Sokol.

  • What it doesn't do :

    • Window creation

      • Use with GLFW, SDL, etc.

    • Input handling

    • Audio, UI, Physics

    • Scene graph

    • "it's not an engine nor even a lightweight framework. it is solely a render and in the way it abstracts the different render libraries (vulkan, gl, metal, android etc) it can be thought of as an rlgl on steroids. There is not even any way to load a "mesh" into bgfx - that code is in the example glue and would frankly be better implemented as part of RayLib."

  • Example :

    • If Vulkan is available and selected as the backend (either manually or automatically), bgfx:

    • Initializes Vulkan instance, physical device, and logical device internally.

    • Manages swapchains, command buffers, pipeline states, and memory using Vulkan APIs.

    • Abstracts Vulkan‚Äôs complexity via its own API layer, hiding most boilerplate.

  • Use :

    • "Guild Wars 2 is using bgfx to switch their renderer from dx9 to dx11. So if it's good enough for a AAA game"

  • State and opinions :

    • Raysan (RayLib author):

      • I've been following bgfx  since the beginning, it's an amazing library, but I have no plans at the moment to replace rlgl . Actually, the latest rlgl  version goes in the line of bgfx , abstracting all Mesh / Material  higher-level info to a models  module and letting rlgl  just manage buffers and ids.

      • Also, one of the things I like most from rlgl  is the OpenGL 1.1 style API, I think it makes the difference, allowing amateur users to easily code in an OpenGL 1.1 style. bgfx  really targets lower-level programming and also more advanced uses.

    • (2025-06-25)

      • No Odin binding.

    • Complaint about complexity and API separation .

      • "20 different things just to load a texture".

      • Does not use CMake.

      • That was 4 years ago.
